## B. Cross-cutting follow-ups (real debt; fix once at the seam)

- [ ] **[security] Capture-layer SSRF denylist — app-wide, pre-existing.** The capture fetch (`page.goto` in
  `capture/url-screenshot.ts`, the readable fetch in `capture/url-readable.ts`) has no private-address/loopback
  denylist (RFC1918, `127.0.0.0/8`, `169.254.169.254`, `::1`). EVERY unauthed create path reaches it:
  `POST /api/collections/:cid/items`, `POST /share` (13.3), `POST /api/v1/items`, and the snapshot capture
  (16.1). Surfaced in the 13.3 review (Winston); NOT introduced by this batch — `/share` is actually stricter
  (it enforces `^https?://`). **Fix:** one denylist at the shared capture-fetch seam (resolve host → reject
  private/loopback before fetching) protects all paths at once. (memory: `board-oss-capture-ssrf`)
- [ ] **[correctness] `deleteItemWithAssets` resolves `snapshots/*` assets under the wrong dir.** It joins every
  asset path under `screenshotsDir` by basename, but Epic 16 snapshot assets live under `snapshotsDir`
  (`db/snapshot-asset.ts`). So deleting an item/board with a `kind='snapshot'` asset won't unlink its `.html`
  (orphaned file). Epic 16 added snapshots without updating the deleter. **Fix:** make `deleteItemWithAssets`
  resolve by asset kind / path-prefix (screenshots vs snapshots dir). Surfaced in the 15.3 review.
- [ ] **[ux/design] Materialized board fields aren't editable (15.3).** The destination descriptor is minimal
  (`fields:[]`), a deliberate v1 choice (no descriptor-merge across heterogeneous sources), so a materialized
  item's notes/favorite are editable but its descriptor FIELDS are not (empty `patchItemFields` allowlist).
  Divergence still holds. **Fix (design):** choose/merge a descriptor for the materialized board so user fields
  stay PATCHable.
- [ ] **[ops] Backfill CLI has no cross-process Chrome lock (16.3).** Concurrency-1 is PER-PROCESS; the
  `archive:backfill` CLI is a second process with its own Chrome. Running it while the server is also capturing
  puts two Chromiums on the box (the OOM NFR-1 prevents in-process). Currently doc-only ("stop the server
  first"). **Fix:** a PID/port lock under `DATA_DIR` (refuse to run if the server is up).
